# 传感器设备录入增强推荐方案

## 背景

当前系统已支持设备的手动录入、批量导入和智能识别功能。本方案针对传感器设备（CO、CO2、温湿度、PM、空气质量传感器）提出增强建议，采用"分层+灵活参数"的设计思路。

## 核心设计思路

### 1. 设备类型分层

使用基础传感器类型作为 `device_type`：
- CO传感器
- CO2传感器
- 温湿度传感器
- PM传感器
- 空气质量传感器

### 2. 灵活参数存储

使用 `key_params`（JSON格式）存储具体检测参数和规格：

```json
{
  "检测对象": {"value": "CO2", "required": true},
  "量程": {"value": "0-2000", "unit": "ppm", "required": true},
  "输出信号": {"value": "4-20mA", "required": true},
  "精度": {"value": "±50", "unit": "ppm", "required": false}
}
```

## 推荐实施方案

### 方案一：配置驱动的动态表单（推荐）

#### 优点
- 灵活性高，易于扩展新传感器类型
- 无需修改代码即可调整参数配置
- 用户体验好，表单自动适配

#### 实施步骤

1. **配置表结构设计**
   ```sql
   CREATE TABLE device_type_params_config (
     id INTEGER PRIMARY KEY,
     device_type TEXT NOT NULL,
     param_name TEXT NOT NULL,
     param_label TEXT,
     param_type TEXT,  -- text, number, select
     unit TEXT,
     required BOOLEAN DEFAULT 0,
     display_order INTEGER
   );
   ```

2. **预置五种传感器配置**
   
   为每种传感器类型配置其特有参数：
   - CO传感器：检测对象、量程、输出信号、精度
   - CO2传感器：检测对象、量程、输出信号、精度
   - 温湿度传感器：检测对象、温度量程、湿度量程、温度精度、湿度精度、输出信号
   - PM传感器：检测对象（PM2.5/PM10）、量程、精度、输出信号
   - 空气质量传感器：检测对象（多选）、各参数量程、各参数精度、输出信号

3. **前端动态表单实现**
   ```javascript
   // 伪代码示例
   async function loadDynamicForm(deviceType) {
     const config = await fetchParamsConfig(deviceType);
     const formFields = config.map(param => ({
       name: param.param_name,
       label: param.param_label,
       type: param.param_type,
       unit: param.unit,
       required: param.required
     }));
     renderForm(formFields);
   }
   ```

4. **权重配置调整**
   
   在现有权重配置中添加：
   ```python
   WEIGHT_CONFIG = {
     'device_type': 20.0,      # 最高优先级
     'key_params': 15.0,       # 关键参数
     'brand': 10.0,
     'spec_model': 5.0
   }
   ```

5. **同义词映射扩展**
   
   在现有同义词配置中添加传感器相关：
   ```python
   SENSOR_SYNONYMS = {
     'CO传感器': ['一氧化碳传感器', 'CO检测器'],
     'CO2传感器': ['二氧化碳传感器', 'CO2检测器'],
     '温湿度传感器': ['温度湿度传感器', '温湿度探头'],
     'PM传感器': ['颗粒物传感器', 'PM2.5传感器', 'PM10传感器'],
     '空气质量传感器': ['空气监测仪', '环境监测传感器']
   }
   ```

6. **白名单特征更新**
   
   在现有白名单中添加传感器关键词：
   ```python
   WHITELIST_FEATURES = [
     '水', '气', '阀',
     'co', 'co2', 'pm2.5', 'pm10',
     '温度', '湿度', '颗粒物'
   ]
   ```

### 方案二：硬编码表单（快速实施）

#### 优点
- 实施快速，无需额外配置表
- 代码逻辑清晰

#### 缺点
- 扩展性差，新增传感器需要修改代码
- 维护成本高

#### 实施步骤

直接在前端组件中为每种传感器类型编写固定表单模板。

**不推荐此方案**，因为系统已有配置管理能力，应充分利用。

## 关键技术点

### 1. key_params 的匹配逻辑

在规则生成器中增强 key_params 的处理：

```python
def extract_key_params_features(key_params):
    """从 key_params 中提取关键特征用于匹配"""
    features = []
    
    if not key_params:
        return features
    
    params = json.loads(key_params) if isinstance(key_params, str) else key_params
    
    for param_name, param_data in params.items():
        # 提取检测对象
        if param_name == '检测对象':
            value = param_data.get('value', '')
            features.append(f"检测对象:{value}")
        
        # 提取量程（标准化处理）
        elif param_name == '量程':
            value = param_data.get('value', '')
            unit = param_data.get('unit', '')
            features.append(f"量程:{value}{unit}")
    
    return features
```

### 2. 设备命名规范

建议设备名称格式：`{品牌} {检测对象} {设备类型} {规格型号}`

示例：
- "霍尼韦尔 CO2 传感器 H7012"
- "西门子 温湿度 传感器 QFA2060"

### 3. 批量导入模板

Excel 模板增加 key_params 列，支持 JSON 格式输入：

| 设备类型 | 品牌 | 规格型号 | key_params |
|---------|------|---------|-----------|
| CO2传感器 | 霍尼韦尔 | H7012 | {"检测对象":{"value":"CO2","required":true},"量程":{"value":"0-2000","unit":"ppm","required":true}} |

或使用展开列的方式：

| 设备类型 | 品牌 | 规格型号 | 检测对象 | 量程 | 量程单位 | 输出信号 | 精度 | 精度单位 |
|---------|------|---------|---------|------|---------|---------|------|---------|
| CO2传感器 | 霍尼韦尔 | H7012 | CO2 | 0-2000 | ppm | 4-20mA | ±50 | ppm |

**推荐使用展开列方式**，用户体验更好。

## 数据验证规则

### 必填字段验证
- 检测对象：必填
- 量程：必填，格式验证（数值范围）
- 输出信号：必填

### 格式验证
- 量程格式：`最小值-最大值` 或单个数值
- 精度格式：支持 `±数值` 格式
- 单位：从预定义列表选择

### 逻辑验证
- 温湿度传感器必须至少包含温度或湿度之一
- 空气质量传感器必须至少包含一个检测对象
- PM传感器的检测对象必须是 PM2.5 或 PM10

## 风险和注意事项

### 数据迁移
- 现有设备数据可能需要迁移到新的 key_params 结构
- 建议提供数据迁移脚本和回滚方案

### 性能考虑
- key_params 的 JSON 解析可能影响性能
- 建议对常用查询添加索引
- 考虑缓存配置数据

### 兼容性
- 确保新功能不影响现有设备类型的录入
- 保持 API 向后兼容

## 总结

推荐采用**方案一：配置驱动的动态表单**，理由如下：

1. **灵活性**：可通过配置快速支持新的传感器类型
2. **可维护性**：配置与代码分离，降低维护成本
3. **用户体验**：动态表单自动适配，减少用户学习成本
4. **扩展性**：为未来支持更多设备类型打下基础

实施时建议分三个阶段逐步推进，先实现核心功能，再逐步完善增强功能和批量导入能力。
