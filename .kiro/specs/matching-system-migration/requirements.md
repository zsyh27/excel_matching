# 匹配系统迁移 - 需求文档

## 1. 项目概述

### 1.1 项目背景

当前系统存在两套设备匹配系统**同时在使用中**:
- **旧系统 (MatchEngine)**: 用于生产环境Excel批量匹配 (`/api/match` 端点)
- **新系统 (IntelligentExtractionAPI)**: 用于智能提取预览和测试 (`/api/intelligent-extraction/*` 端点)

新系统在设计上更符合业务需求,采用**五步流程**(设备类型识别→参数提取→辅助信息提取→智能匹配→UI展示),匹配准确率更高。现在需要将**所有匹配逻辑**完全切换到新系统,统一所有匹配场景的处理逻辑。

**关键发现**:
- ✅ **前端已完成**: Excel导入、设备行选取、多候选列表显示、手动选择功能都已开发完成
- ✅ **新系统已实现**: 五步流程API已完整实现 (`IntelligentExtractionAPI`)
- ✅ **前端无需改动**: 页面端几乎不需要做任何改动
- 🎯 **核心工作**: 只需要在后台处理逻辑上进行切换,将 `/api/match` 从使用 `MatchEngine` 改为使用 `IntelligentExtractionAPI`
- 🎯 **测试页面调整**: http://localhost:3000/testing 中的六步流程实时预览需要删除步骤0（文本预处理），改为五步流程实时预览
- 🎯 **归一化时机**: 归一化/同义词映射应在步骤1-3提取结果之后应用，而不是在提取之前

### 1.2 项目目标

将 `/api/match` 端点从旧系统(MatchEngine)迁移到新系统(IntelligentExtractionAPI),实现:
1. 统一的五步流程处理逻辑（删除步骤0文本预处理）
2. 更高的匹配准确率(设备类型识别准确率100%)
3. 保持前端兼容性(返回格式不变)
4. 保持性能指标(响应时间<500ms)
5. 完全移除旧系统依赖
6. 调整测试页面UI（六步改为五步）
7. 归一化/同义词映射在步骤1-3提取后应用

### 1.3 项目范围

**包含范围**(核心工作):
- ✅ 修改 `/api/match` 端点,使用 `IntelligentExtractionAPI` 替代 `MatchEngine`

## 3. 功能需求

### 3.1 核心迁移工作

#### 3.1.1 `/api/match` 端点迁移 ⭐ 核心任务

**需求描述**: 将 `/api/match` 端点从使用 `MatchEngine` 改为使用 `IntelligentExtractionAPI`

**当前实现**:
```python
# 当前使用 MatchEngine
match_result, cache_key = match_engine.match(
    features=features,
    input_description=original_description,
    record_detail=record_detail
)

# 获取候选设备列表
all_candidates = match_engine._evaluate_all_candidates(features)
```

**目标实现**:
```python
# 改为使用 IntelligentExtractionAPI
match_result = intelligent_extraction_api.match(
    text=original_description,
    top_k=20
)

# 从新系统获取候选设备列表
candidates_list = match_result['data']['candidates']
```

**详细说明**:
- 保持API端点路径不变 (`POST /api/match`)
- 保持请求格式不变 (接收 `rows` 数组)
- 保持响应格式不变 (返回 `matched_rows` 和 `candidates` 列表)
- 内部逻辑从 `MatchEngine` 切换到 `IntelligentExtractionAPI`
- 返回前20个候选设备(与当前行为一致)

**验收标准**:
- [ ] `/api/match` 端点成功调用 `IntelligentExtractionAPI`
- [ ] 返回格式与前端兼容(包含 `candidates` 数组)
- [ ] 每个候选包含: `device_id`, `matched_device_text`, `unit_price`, `match_score`
- [ ] 支持 `detail_cache_key` (如果新系统支持)
- [ ] 响应时间 < 500ms
- [ ] 前端无需任何改动即可正常工作

### 3.2 性能优化

#### 3.2.1 设备类型索引缓存

**需求描述**: 在 `IntelligentMatcher` 中添加设备类型索引缓存,加速设备过滤

**详细说明**:
- 在初始化时构建 `device_cache_by_type` 索引
- 根据识别的设备类型快速过滤候选设备
- 减少不必要的匹配计算

**验收标准**:
- [ ] 设备类型索引构建成功
- [ ] 匹配速度提升30%以上
- [ ] 内存占用增加 < 100MB

### 3.3 兼容性保证

#### 3.3.1 返回格式兼容

**需求描述**: 确保新系统返回格式与前端完全兼容

**详细说明**:
- 候选设备列表格式: `[{device_id, matched_device_text, unit_price, match_score, brand, device_name, spec_model}]`
- 匹配结果格式: `{device_id, matched_device_text, unit_price, match_status, match_score, match_reason}`
- 统计信息格式: `{total_devices, matched, unmatched, accuracy_rate}`

**验收标准**:

- [ ] 前端无需修改即可正常显示匹配结果

- **MatchEngine**: 旧的匹配引擎系统(基于权重的特征匹配)
