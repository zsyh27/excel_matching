# 设备录入阶段独立特征提取完成报告

## 问题描述

用户反馈的问题:
1. 规格型号"HST-RA"被截断为"hst-r" (单位"a"被错误删除)
2. 规格型号被识别为"参数"(权重1)而不是"型号"(权重5)
3. 设备录入阶段的数据处理逻辑与匹配阶段混在一起,导致各种问题

## 根本原因

设备录入阶段使用了与匹配阶段相同的`TextPreprocessor`,该预处理器包含:
- 复杂的文本拆分逻辑
- 单位删除逻辑(导致"HST-RA"中的"A"被删除)
- 智能拆分逻辑
- 元数据关键词处理
- 括号提取逻辑

这些逻辑是为匹配阶段的非结构化文本设计的,不适合设备录入阶段的结构化数据。

## 解决方案

### 1. 创建独立的设备特征提取器

创建了`DeviceFeatureExtractor`类(`backend/modules/device_feature_extractor.py`),专门用于设备录入阶段:

**设计原则:**
1. **直接映射** - 设备字段直接作为特征,不做复杂解析
2. **不做拆分** - 保持字段完整性,不使用分隔符拆分
3. **不删除单位** - 保持原始数据完整性
4. **简单归一化** - 只做基本的大小写转换和去空格
5. **明确类型** - 每个特征都有明确的类型标记(brand, device_type, device_name, model, parameter)
6. **固定权重** - 根据字段类型直接分配权重,不使用关键词判断

**特征提取规则:**
```python
# 1. brand → 品牌特征 (权重10)
# 2. device_type → 设备类型特征 (权重20)
# 3. device_name → 设备名称特征 (权重1)
# 4. spec_model → 型号特征 (权重5)
# 5. key_params → 关键参数特征 (权重15)
```

### 2. 修改规则生成器

修改了`RuleGenerator`类(`backend/modules/rule_generator.py`):
- 移除了对`TextPreprocessor`的依赖
- 使用`DeviceFeatureExtractor`进行特征提取
- 简化了构造函数,只需要`config`参数
- 删除了复杂的权重分配逻辑

### 3. 更新DataLoader

更新了`DataLoader`中初始化`RuleGenerator`的代码:
```python
# 旧版本
rule_generator = RuleGenerator(self.preprocessor, default_threshold)

# 新版本
rule_generator = RuleGenerator(config, default_threshold)
```

### 4. 保持匹配阶段不变

匹配阶段仍然使用`TextPreprocessor`,因为需要处理非结构化的用户输入文本。

**关键改进:**
- 规格型号"HST-RA"保持完整,不被截断为"hst-r"
- 规格型号被正确识别为"model"类型,权重为5
- 所有特征都有明确的类型和权重

## 架构改进

### 之前的架构(有问题)

```
设备录入阶段 ──┐
              ├──> TextPreprocessor (复杂逻辑)
匹配阶段 ─────┘
```

### 现在的架构(正确)

```
设备录入阶段 ──> DeviceFeatureExtractor (简单直接)
匹配阶段 ─────> TextPreprocessor (复杂逻辑)
```

## 设计原则对比

| 方面 | 设备录入阶段 | 匹配阶段 |
|------|------------|---------|
| 数据类型 | 结构化数据 | 非结构化文本 |
| 提取方式 | 直接映射 | 智能解析 |
| 是否拆分 | 否 | 是 |
| 单位处理 | 保留 | 删除 |
| 归一化 | 简单(小写) | 复杂(映射+转换) |
| 类型判断 | 明确(字段类型) | 推断(关键词) |
| 权重分配 | 固定(配置) | 智能(规则) |
