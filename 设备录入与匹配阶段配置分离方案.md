# 设备录入与匹配阶段配置分离方案

## 问题分析

### 当前问题

系统存在两个不同的使用场景:

1. **设备录入阶段** (`mode='device'`)
   - 处理设备库中的标准化数据
   - 数据格式规范,字段清晰
   - 需要完整保留设备类型、设备名称等关键信息
   - 目标:生成准确的匹配规则

2. **匹配阶段** (`mode='matching'`)
   - 处理Excel中的混乱输入
   - 数据格式不规范,可能包含字段名前缀
   - 需要智能清理和特征提取
   - 目标:从混乱文本中提取有效特征进行匹配

### 根本原因

当前系统的**元数据关键词**和**特征白名单**配置是为**匹配阶段**设计的,用于处理混乱的Excel输入。但这些配置也被应用到了**设备录入阶段**,导致:

- "温度"在元数据关键词中 → "温度传感器"被拆分成"传感器"
- 设备类型权重从20降到1
- 匹配准确度下降

## 解决方案

### 方案1: 在设备录入阶段跳过元数据关键词处理(推荐)

**核心思想**: 在`mode='device'`时,跳过元数据关键词的前缀删除逻辑。

#### 实现步骤

1. **修改`_extract_bracket_features`方法**,添加`mode`参数
2. **在`mode='device'`时跳过元数据关键词前缀删除**
3. **保持其他处理逻辑不变**

#### 代码修改


#### 优点

- ✅ 最小化代码修改
- ✅ 不影响现有匹配逻辑
- ✅ 设备录入阶段不受元数据关键词影响
- ✅ 向后兼容

#### 缺点

- ⚠️ 元数据关键词配置仍然可能被误用
- ⚠️ 用户可能不理解为什么配置在设备录入时不生效

### 方案2: 分离配置(更彻底)

**核心思想**: 为设备录入和匹配阶段创建独立的配置。

#### 配置结构

```json
{
  "metadata_keywords": {
    "device_mode": [],  // 设备录入阶段使用(通常为空)
    "matching_mode": [  // 匹配阶段使用
      "型号", "品牌", "规格", "参数", "温度", "压力"
    ]
  },
  "intelligent_extraction": {
    "feature_quality_scoring": {
      "whitelist_features": {
        "device_mode": [],  // 设备录入阶段使用(通常为空)
        "matching_mode": ["水", "气", "阀"]  // 匹配阶段使用
      }
    }
  }
}
```

#### 优点

- ✅ 配置职责清晰
- ✅ 用户可以分别管理两个阶段的配置
- ✅ 更灵活

#### 缺点

- ⚠️ 需要修改配置结构
- ⚠️ 需要修改前端UI
- ⚠️ 需要数据迁移
- ⚠️ 实现复杂度较高

### 方案3: 使用黑名单机制

**核心思想**: 在设备录入阶段,维护一个"不应该删除的前缀"黑名单。

#### 配置结构

```json
{
  "metadata_keywords": [
    "型号", "品牌", "规格", "参数", "温度", "压力"
  ],
  "metadata_keywords_device_mode_blacklist": [
    "温度", "压力", "湿度", "流量"  // 设备录入时不删除这些前缀
  ]
}
```

#### 优点

- ✅ 配置结构变化小
- ✅ 向后兼容
- ✅ 实现简单

#### 缺点

- ⚠️ 需要维护额外的黑名单
- ⚠️ 逻辑略显复杂

## 推荐方案

**推荐使用方案1: 在设备录入阶段跳过元数据关键词处理**

理由:
1. 实现简单,风险低
2. 不需要修改配置结构
3. 不需要修改前端UI
4. 向后兼容
5. 解决了当前的核心问题

## 实施计划

### 第一阶段: 快速修复(方案1)

1. 修改`_extract_bracket_features`方法,添加`mode`参数
2. 在`mode='device'`时跳过元数据关键词处理
3. 测试验证
4. 重新生成所有设备规则

### 第二阶段: 长期优化(可选)

如果未来需要更灵活的配置管理,可以考虑:
1. 实施方案2(分离配置)

3. 提供配置迁移工具

## 测试验证

### 测试用例

1. **设备录入阶段测试**
   ```python
   # mode='device'
   输入: "温度传感器"
   期望输出: ["温度传感器"]  # 不被拆分
   
   输入: "压力传感器"
   期望输出: ["压力传感器"]  # 不被拆分
   ```

2. **匹配阶段测试**
   ```python
   # mode='matching'
   输入: "温度:25℃"
   期望输出: ["25℃"]  # 删除"温度"前缀
   
   输入: "型号:QAA2061"
   期望输出: ["QAA2061"]  # 删除"型号"前缀
   ```

3. **规则生成测试**
   ```python
   设备: 霍尼韦尔 温度传感器 HST-RA
   期望特征: ["霍尼韦尔", "温度传感器", ...]
   期望权重: 温度传感器=20
   ```

## 影响评估

### 受影响的组件

1. `backend/modules/text_preprocessor.py`
   - `_extract_bracket_features` 方法
   - `_extract_features_with_detail` 方法

2. `backend/modules/rule_generator.py`
   - 已经使用`mode='device'`,无需修改

3. 数据库
   - 需要重新生成所有设备规则

### 不受影响的组件

1. 前端UI - 无需修改
2. 配置结构 - 无需修改
3. 匹配引擎 - 无需修改

## 回滚计划

如果修复后出现问题:

1. 恢复`_extract_bracket_features`方法的原始代码
2. 从数据库备份恢复规则表
3. 验证系统功能正常
