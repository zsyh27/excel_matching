## 系统概述

本系统提供了完整的设备信息录入、管理和匹配功能，支持手动录入、批量导入和智能识别等多种方式。系统采用三层架构：

- **配置层**：`data/static_config.json` - 存储系统配置、权重配置、白名单等
- **代码层**：Python后端服务 - 实现业务逻辑
- **数据层**：数据库/JSON文件 - 存储设备和规则数据

### 核心功能

1. **设备库管理**：维护标准设备信息和价格
2. **智能匹配**：自动匹配上传的设备清单
3. **规则管理**：为每个设备自动生成匹配规则
4. **配置管理**：调整系统匹配参数
5. **批量操作**：支持批量导入、导出、规则生成

## 设备数据模型

设备信息包含以下核心字段：

```python
@dataclass
class Device:
    # 基础字段（必填）
    device_id: str          # 设备唯一ID（自动生成，格式：D_yyyyMMddHHmmss_随机数）
    brand: str              # 品牌
    device_name: str        # 设备名称
    spec_model: str         # 规格型号
    detailed_params: str    # 详细参数（文本格式，用于向后兼容）
    unit_price: float       # 不含税单价
    
    # 新增字段（可选）
    device_type: Optional[str] = None           # 设备类型（如"电动调节阀"）
    key_params: Optional[Dict[str, Any]] = None # 关键参数JSON（结构化数据）
    raw_description: Optional[str] = None       # 原始描述文本
    confidence_score: Optional[float] = None    # 置信度评分（0-100）
    input_method: str = 'manual'                # 录入方式（manual/batch/intelligent）
    created_at: Optional[datetime] = None       # 创建时间
    updated_at: Optional[datetime] = None       # 更新时间
```

### key_params vs detailed_params

这是系统中两个重要的参数存储字段，理解它们的区别对于正确使用系统至关重要：

| 字段 | 数据类型 | 优先级 | 用途 | 权重 | 示例 |
|------|----------|--------|------|------|------|
| **key_params** | JSON | **高（优先使用）** | 结构化存储关键参数 | 15.0 | `{"通径":{"value":"DN15"},"介质":{"value":"水"}}` |
| **detailed_params** | Text | 低（回退使用） | 文本描述，向后兼容 | 较低 | `"介质: 水\n通径: DN15\n通数: 二通"` |

**特征提取优先级**：
1. ✅ 优先从 `key_params` 提取特征（结构化，准确性高，权重15.0）
2. ⚠️ 当 `key_params` 为空时，回退到 `detailed_params`（文本解析，权重较低）

**建议**：
- 新录入设备优先使用 `key_params`
- 两个字段可以同时填写，系统会优先使用 `key_params`
- 如果只填写 `detailed_params`，系统会自动从中提取特征（作为回退机制）

## 数据存储架构

系统支持两种存储模式，可通过配置切换：

### 模式一：JSON文件存储（默认模式）

**文件位置**：
- 设备数据：`data/static_device.json`
- 规则数据：`data/static_rule.json`
- 配置数据：`data/static_config.json`

**优点**：
- 简单易用，无需数据库
- 易于调试和查看
- 适合小规模数据

**缺点**：
- 并发性能较低
- 不支持复杂查询
- 数据量大时性能下降

### 模式二：数据库存储（推荐模式）

**数据库文件**：`data/devices.db`（SQLite）

**数据库表结构**：

#### 1. devices 表（设备信息表）

| 字段名 | 类型 | 说明 | 是否必填 |
|--------|------|------|----------|
| device_id | 字符串 | 设备唯一标识（主键） | 是 |
| brand | 字符串 | 品牌名称 | 是 |
| device_name | 字符串 | 设备名称 | 是 |
| spec_model | 字符串 | 规格型号 | 是 |
| device_type | 字符串 | 设备类型 | 否 |
| unit_price | 浮点数 | 单价（元） | 是 |
| detailed_params | 文本 | 详细参数（文本格式） | 否 |
| key_params | JSON | 关键参数（JSON格式） | 否 |
| raw_description | 文本 | 原始描述文本 | 否 |
| confidence_score | 浮点数 | 置信度评分 | 否 |
| input_method | 字符串 | 录入方式 | 否 |
| created_at | 日期时间 | 创建时间 | 否 |
| updated_at | 日期时间 | 更新时间 | 否 |

#### 2. rules 表（匹配规则表）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| rule_id | 字符串 | 规则唯一标识（主键） |
| target_device_id | 字符串 | 关联的设备ID（外键） |
| auto_extracted_features | JSON | 自动提取的特征列表 |
| feature_weights | JSON | 特征权重配置 |
| match_threshold | 浮点数 | 匹配阈值 |
| remark | 文本 | 备注说明 |

**优点**：
- 高性能，支持复杂查询
- 自动事务管理
- 支持级联删除
- 支持批量操作
- 数据一致性好

**缺点**：
- 需要数据库迁移
- 相对复杂

### 切换存储模式

在 `config.py` 中设置 `STORAGE_MODE`：

```python
STORAGE_MODE = 'database'  # 或 'json'
```

**注意**：切换存储模式需要迁移数据。

## 规则生成机制

系统会为每个设备自动生成匹配规则，规则生成分为三个步骤。设备录入阶段使用专门的 `DeviceFeatureExtractor` 进行特征提取，与匹配阶段的逻辑完全分离。

### 设备录入阶段 vs 匹配阶段

| 特性 | 设备录入阶段 | 匹配阶段 |
|------|------------|---------|
| 数据来源 | 结构化表单输入 | 非结构化Excel文本 |
| 特征提取器 | DeviceFeatureExtractor | TextPreprocessor |
| 处理方式 | 直接映射，保持完整性 | 智能解析，拆分清理 |
| 单位处理 | 保留（如"HST-RA"保持完整） | 删除（如"25℃"变为"25"） |
| 文本拆分 | 不拆分（如"温度传感器"保持完整） | 智能拆分（如"温度传感器"可能拆分） |
| 类型判断 | 明确字段类型 | 关键词推断 |
| 元数据关键词 | 不使用 | 使用（删除"型号:"等前缀） |
| 设备类型关键词 | 不使用 | 使用（识别设备类型） |

### 步骤1：特征提取（extract_features）

**设计原则**（DeviceFeatureExtractor）：

1. **直接映射** - 设备字段直接作为特征，不做复杂处理
2. **不做拆分** - 保持字段完整性（如"温度传感器"不拆分）
3. **不删除单位** - 保持原始数据（如"HST-RA"的"A"不删除）
4. **简单归一化** - 只做大小写转换和空格处理
5. **明确类型** - 每个特征都有明确的类型标记（brand、device_type、model、parameter）

**提取顺序和权重**：

1. **品牌（brand）** → 品牌特征（权重10）
2. **设备类型（device_type）** → 设备类型特征（权重20）
3. **设备名称（device_name）** → 设备名称特征（权重1）
4. **规格型号（spec_model）** → 型号特征（权重5）
5. **关键参数（key_params）** → 关键参数特征（权重15）

**示例**：

```python
# 输入设备
device = Device(
    device_id="D_001",
    brand="霍尼韦尔",
    device_name="室内温度传感器",
    device_type="温度传感器",
    spec_model="HST-RA",
    key_params={
        "温度量程": {"value": "-20~60℃"},
        "温度信号类型": {"value": "NTC 10K"},
        "安装位置": {"value": "室内墙装"}
    }
)

# 提取的特征（DeviceFeatureExtractor）
features = [
    DeviceFeature(feature="霍尼韦尔", type="brand", weight=10.0, source="brand"),
    DeviceFeature(feature="温度传感器", type="device_type", weight=20.0, source="device_type"),
    DeviceFeature(feature="室内温度传感器", type="device_name", weight=1.0, source="device_name"),
    DeviceFeature(feature="hst-ra", type="model", weight=5.0, source="spec_model"),
    DeviceFeature(feature="-20~60℃", type="parameter", weight=15.0, source="key_params.温度量程"),
    DeviceFeature(feature="ntc 10k", type="parameter", weight=15.0, source="key_params.温度信号类型"),
    DeviceFeature(feature="室内墙装", type="parameter", weight=15.0, source="key_params.安装位置")
]
```

**关键改进**：

✅ **规格型号保持完整**：
- 修复前："HST-RA" → "hst-r"（A被删除）
- 修复后："HST-RA" → "hst-ra"（保持完整）

✅ **设备类型不拆分**：
- 修复前："温度传感器" → ["传感器"]（温度被删除）
- 修复后："温度传感器" → ["温度传感器"]（保持完整）

✅ **权重准确分配**：
- 修复前："hst-r"权重1.0（被识别为参数）
- 修复后："hst-ra"权重5.0（正确识别为型号）

### 步骤2：权重分配（assign_weights）

**设备录入阶段权重分配原则**：

设备录入阶段的权重分配**只使用设备字段类型**，不使用关键词判断：

| 优先级 | 特征类型 | 默认权重 | 判断方法 | 配置键 |
|--------|----------|----------|----------|--------|
| 1️⃣ 最高 | 设备类型（device_type字段） | 20.0 | 直接来自device_type字段 | `device_type_weight` |
| 2️⃣ 次高 | key_params参数特征 | 15.0 | 来自key_params的特征 | `key_params_weight` |
| 3️⃣ 中高 | 品牌（brand字段） | 10.0 | 直接来自brand字段 | `brand_weight` |
| 4️⃣ 中 | 规格型号（spec_model字段） | 5.0 | 直接来自spec_model字段 | `model_weight` |
| 5️⃣ 低 | 设备名称/通用参数 | 1.0 | 来自device_name或其他 | `parameter_weight` |

**不使用的判断逻辑**（仅在匹配阶段使用）：

- ❌ 设备类型关键词（device_type_keywords）
- ❌ 品牌关键词（brand_keywords）
- ❌ 元数据关键词（metadata_keywords）
- ❌ 特征白名单（whitelist_features）
- ❌ 型号模式匹配（字母+数字组合）

**权重配置**（在 `static_config.json` 中）：

```json
{
  "feature_weight_config": {
    "brand_weight": 10,
    "device_type_weight": 20,
    "key_params_weight": 15,
    "model_weight": 5,
    "parameter_weight": 1
  }
}
```

**权重分配示例**：

```python
# 特征权重分配（设备录入阶段）
feature_weights = {
    "霍尼韦尔": 10.0,          # 来自brand字段
    "温度传感器": 20.0,        # 来自device_type字段
    "室内温度传感器": 1.0,     # 来自device_name字段
    "hst-ra": 5.0,            # 来自spec_model字段
    "-20~60℃": 15.0,          # 来自key_params参数
    "ntc 10k": 15.0,          # 来自key_params参数
    "室内墙装": 15.0           # 来自key_params参数
}
```

**重要说明**：

1. **权重分配只看字段来源**
   - 不使用关键词匹配
   - 不使用模式识别
   - 直接根据字段类型分配权重

2. **设备类型保持完整**
   - "温度传感器"作为一个整体，权重20.0
   - 不会被拆分为"温度"和"传感器"

3. **规格型号保持完整**
   - "HST-RA"保持完整，权重5.0
   - 不会删除单位后缀

4. **key_params参数权重统一**
   - 所有来自key_params的特征权重都是15.0
   - 不区分参数类型

### 步骤3：阈值设置

- **默认匹配阈值**：5.0
- **配置位置**：`global_config.default_match_threshold`
- **匹配规则**：总权重分数 >= 阈值 才认为匹配成功

**阈值说明**：
- 匹配阈值是累计权重阈值，不是百分比
- 匹配得分 = 所有匹配特征的权重之和
- 只有当匹配得分 ≥ 阈值时，才认为匹配成功

**阈值计算示例**：

假设匹配到以下特征：
- 品牌关键词"霍尼韦尔"：10.0
- 设备类型关键词"传感器"：15.0
- 型号"qaa2061"：5.0
- 总得分：30.0

如果阈值设置为5.0，则匹配成功（30.0 ≥ 5.0）

### 配置说明

#### 设备录入阶段配置（仅影响设备库规则生成）

**feature_weight_config**（特征权重配置）：

```json
{
  "feature_weight_config": {
    "brand_weight": 10,           // 品牌字段权重
    "device_type_weight": 20,     // 设备类型字段权重
    "key_params_weight": 15,      // key_params参数权重
    "model_weight": 5,            // 规格型号字段权重
    "parameter_weight": 1         // 通用参数权重
  }
}
```

**device_params**（设备参数配置）：

- **配置路径**：`backend/config/device_params.yaml`
- **作用**：定义每种设备类型有哪些参数
- **用途**：动态表单生成、参数验证
- **示例**：温度传感器有"温度量程"、"温度信号类型"等参数

#### 匹配阶段配置（仅影响Excel匹配）

**device_type_keywords**（设备类型关键词）：

- **配置路径**：`device_type_keywords`
- **作用**：识别用户输入中的设备类型
- **用途**：从混乱的Excel文本中识别设备类型
- **示例**："温度"、"传感器"、"阀门"等

**metadata_keywords**（元数据关键词）：

- **配置路径**：`metadata_keywords`
- **作用**：识别字段名称（如"型号"、"通径"、"介质"）
- **处理**：在特征提取时，会删除这些前缀，只保留值部分
- **示例**："介质: 水" → "水"
- **注意**：仅在匹配阶段使用，设备录入阶段不使用

**whitelist_features**（特征白名单）：

- **配置路径**：`intelligent_extraction.feature_quality_scoring.whitelist_features`
- **作用**：白名单中的特征不受质量评分限制，直接通过过滤
- **用途**：保留重要但可能被过滤的特征（如"水"、"气"、"阀"）
- **注意**：仅在匹配阶段使用，设备录入阶段不使用

**配置示例**：

```json
{
  "intelligent_extraction": {
    "feature_quality_scoring": {
      "enabled": true,
      "min_quality_score": 50,
      "whitelist_features": ["水", "气", "阀"]
    }
  },
  "metadata_keywords": ["型号", "通径", "介质", "温度", "压力"],
  "device_type_keywords": ["传感器", "阀门", "控制器"]
}
```

#### 配置职责划分

| 配置项 | 用途 | 阶段 | 说明 |
|--------|------|------|------|
| device_params | 定义设备结构 | 设备录入 | 指定每种设备类型有哪些参数 |
| feature_weight_config | 分配特征权重 | 设备录入 | 定义不同类型特征的权重 |
| device_type_keywords | 识别设备类型 | 匹配 | 识别用户输入中的设备类型 |
| metadata_keywords | 处理混乱输入 | 匹配 | 删除字段名前缀 |
| whitelist_features | 保护重要特征 | 匹配 | 保护重要但简短的特征 |

## 数据访问层

### DataLoader（统一数据加载器）

**职责**：
- 根据配置选择存储模式（数据库或JSON）
- 提供统一的数据访问接口
- 实现存储模式自动回退机制

**初始化方式**：

```python
# 新方式（推荐）
data_loader = DataLoader(
    config=Config,
    preprocessor=preprocessor
)

# 传统方式（向后兼容）
data_loader = DataLoader(
    device_file='data/static_device.json',
    rule_file='data/static_rule.json',
    config_file='data/static_config.json',
    preprocessor=preprocessor
)
```

**主要方法**：
- `load_devices()` - 加载所有设备
- `load_rules()` - 加载所有规则
- `get_device_by_id(device_id)` - 根据ID查询设备
- `load_config()` - 加载配置
- `validate_data_integrity()` - 验证数据完整性
- `get_storage_mode()` - 获取当前存储模式

### DatabaseLoader（数据库加载器）

**职责**：
- 从数据库加载设备和规则数据
- 提供设备的CRUD操作
- 将ORM模型转换为数据类

**主要方法**：

**基础操作**：
- `load_devices()` - 从数据库加载所有设备
- `load_rules()` - 从数据库加载所有规则
- `get_device_by_id(device_id)` - 根据ID查询设备
- `add_device(device)` - 添加设备
- `update_device(device)` - 更新设备
- `delete_device(device_id)` - 删除设备（级联删除规则）
- `save_rule(rule)` - 保存或更新规则
- `delete_rule(rule_id)` - 删除规则

**数据一致性检查**：
- `find_devices_without_rules()` - 查找没有规则的设备
- `find_orphan_rules()` - 查找孤立规则（target_device_id不存在）
- `check_data_consistency()` - 执行完整的数据一致性检查
- `fix_consistency_issues()` - 修复数据一致性问题

**批量操作**：
- `batch_add_devices(devices, batch_size, auto_generate_rule)` - 批量添加设备
- `batch_generate_rules(device_ids, force_regenerate)` - 批量生成规则

### JSONLoader（JSON文件加载器）

**职责**：
- 从JSON文件加载设备表、规则表、配置文件
- 验证数据完整性
- 自动生成特征
- 同步规则表与设备表

**主要方法**：
- `load_devices()` - 加载设备表
- `load_rules()` - 加载规则表
- `load_config()` - 加载配置文件
- `validate_data_integrity()` - 验证数据完整性
- `auto_generate_features(device)` - 自动生成设备特征
- `auto_sync_rules_with_devices()` - 自动同步规则表与设备表

### ConfigManager（配置管理器）

**职责**：
- 加载和管理配置文件
- 支持配置热加载（自动检测文件变化）
- 提供配置更新功能

**主要方法**：
- `get_config()` - 获取配置（自动检测更新）
- `update_config(updates)` - 更新配置并保存

## API接口

### 设备管理接口

## 注意事项

### 1. 设备ID管理
- 设备ID会自动生成，格式：`D_yyyyMMddHHmmss_随机数`
- 不要手动修改设备ID
- 设备ID在导出/导入时会被排除

### 2. 必填字段
- 品牌（brand）
- 设备名称（device_name）
- 规格型号（spec_model）
- 单价（unit_price）

### 3. 批量导入
- 批量导入时会自动去重（基于device_id）
- 如果设备已存在，会更新设备信息
- 批量导入会自动生成规则（如果auto_generate_rule=true）

### 4. 规则生成
- 修改设备信息后需要重新生成规则
- 修改配置文件后需要重启服务并重新生成规则
- 规则生成失败不会影响设备保存

### 5. 数据一致性
- 删除设备会同时删除关联的规则（级联删除）
- 定期检查数据一致性（POST /api/consistency/check）
- 修复数据一致性问题（POST /api/consistency/fix）

### 6. 配置修改
- 修改配置文件后必须重启后端服务
- 重启前清除Python `__pycache__`
- 重启后重新生成所有设备规则

### 7. 特征提取优先级
- 优先从key_params提取特征（结构化，准确性高）
- 当key_params为空时，回退到detailed_params（文本解析）
- 建议使用key_params存储关键参数

### 8. 白名单功能
- 白名单特征不受质量评分限制（仅在匹配阶段）
- 用于保留重要但可能被过滤的特征
- 配置路径：`intelligent_extraction.feature_quality_scoring.whitelist_features`

### 9. 权重分配
- 设备录入阶段：只使用字段类型分配权重
- 匹配阶段：使用关键词判断和模式匹配
- 权重配置在`feature_weight_config`中

### 10. 存储模式切换
- 切换存储模式需要迁移数据
- 数据库模式性能更好，推荐使用
- JSON模式简单易用，适合小规模数据

### 11. 架构改进说明

#### 设备录入阶段独立特征提取

**问题**：规格型号被截断（如"HST-RA" → "hst-r"）

**解决方案**：
- 创建独立的`DeviceFeatureExtractor`类
- 不使用`TextPreprocessor`的复杂逻辑
- 保持设备数据的完整性和准确性

**效果**：
- ✅ 规格型号保持完整
- ✅ 权重分配准确
- ✅ 设备类型不拆分

#### 配置分离（mode参数）

**问题**：元数据关键词影响设备录入阶段

**解决方案**：
- 通过`mode`参数区分两个阶段
- `mode='device'` - 设备录入阶段，跳过元数据关键词处理
- `mode='matching'` - 匹配阶段，应用元数据关键词处理

**效果**：
- ✅ 设备类型保持完整
- ✅ 用户可以自由配置元数据关键词
- ✅ 两个阶段互不干扰

#### 权重分配优化

**问题**：设备录入阶段使用了设备类型关键词判断

**解决方案**：
- 移除设备类型关键词判断
- 只使用设备字段类型分配权重
- 权重分配更简单直接

**效果**：
- ✅ 权重分配更准确
- ✅ 配置职责更清晰
- ✅ 代码更易维护

## 常见问题解答

### Q1: key_params和detailed_params有什么区别？应该用哪个？

**A**: 
- **key_params（推荐）**：
  - 数据类型：JSON格式
  - 优先级：高（优先使用）
  - 权重：15.0
  - 用途：结构化存储关键参数
  - 示例：`{"量程": {"value": "0-2000", "unit": "ppm"}}`
  
- **detailed_params（向后兼容）**：
  - 数据类型：文本格式
  - 优先级：低（仅当key_params为空时使用）
  - 权重：较低
  - 用途：文本描述，向后兼容旧数据
  - 示例：`"量程:0-2000ppm\n输出:4-20mA"`

**建议**：
- 新录入的设备优先使用key_params
- 两个字段可以同时填写，系统会优先使用key_params
- 如果只填写detailed_params，系统会自动从中提取特征（作为回退机制）

### Q2: 为什么规格型号"HST-RA"在特征列表中显示为"hst-r"？

**A**: 
这个问题已经修复。根本原因是设备录入阶段使用了与匹配阶段相同的`TextPreprocessor`。

**问题原因**：
- 规格型号中的单位后缀被删除（"HST-RA"中的"A"被识别为单位"a"并删除）
- 结构化数据被当作非结构化文本处理，应用了不必要的拆分和清理逻辑

**修复方案**：
- 创建了独立的`DeviceFeatureExtractor`类，专门用于设备录入阶段
- 设计原则：直接映射、不做拆分、不删除单位、简单归一化
- 修改位置：`backend/modules/device_feature_extractor.py`

**验证结果**：
- 修复前："HST-RA" → "hst-r"（权重1.0，被识别为参数）
- 修复后："HST-RA" → "hst-ra"（权重5.0，正确识别为型号）

### Q3: 为什么设备类型"温度传感器"被拆分了？

**A**: 
这个问题已经修复。根本原因是元数据关键词配置影响了设备录入阶段。

**问题原因**：
- 元数据关键词（如"温度"、"压力"）是为匹配阶段设计的，用于处理混乱的Excel输入
- 但这些配置也被应用到了设备录入阶段，导致设备类型被错误拆分

**修复方案**：
- 在`TextPreprocessor`中通过`mode`参数区分两个阶段
- `mode='device'` - 设备录入阶段，跳过元数据关键词处理
- `mode='matching'` - 匹配阶段，应用元数据关键词处理
- 修改位置：`backend/modules/text_preprocessor.py`

**验证结果**：
- 修复前：设备录入时"温度传感器" → ["传感器"]（温度被删除）
- 修复后：设备录入时"温度传感器" → ["温度传感器"]（保持完整）
- 匹配阶段：仍然可以智能拆分混乱输入

### Q4: 设备录入阶段和匹配阶段有什么区别？

**A**: 
这是系统的核心设计原则，两个阶段使用完全不同的处理逻辑：

**设备录入阶段**：
- 数据来源：结构化表单输入
- 特征提取器：DeviceFeatureExtractor
- 处理方式：直接映射，保持完整性
- 单位处理：保留（如"HST-RA"保持完整）
- 文本拆分：不拆分（如"温度传感器"保持完整）
- 类型判断：明确字段类型
- 配置使用：feature_weight_config、device_params

**匹配阶段**：
- 数据来源：非结构化Excel文本
- 特征提取器：TextPreprocessor
- 处理方式：智能解析，拆分清理
- 单位处理：删除（如"25℃"变为"25"）
- 文本拆分：智能拆分（如"温度传感器"可能拆分）
- 类型判断：关键词推断
- 配置使用：device_type_keywords、metadata_keywords、whitelist_features

- 两个阶段互不干扰，职责清晰

### 重要概念说明

#### 1. 匹配得分计算

**计算公式**：
```
匹配得分 = Σ(匹配特征的权重)
```

**示例**：
假设输入描述："霍尼韦尔CO2传感器 QAA2061"

提取的特征和权重：
- "霍尼韦尔"（品牌关键词）：10.0
- "co2"（设备类型关键词）：15.0
- "传感器"（设备类型关键词）：15.0
- "qaa2061"（型号特征）：5.0

总得分 = 10.0 + 15.0 + 15.0 + 5.0 = 45.0

如果阈值设置为5.0，则匹配成功（45.0 ≥ 5.0）

#### 2. 特征提取优先级

系统按以下顺序提取特征（设备录入阶段）：
1. 品牌（brand）→ 权重10
2. 设备类型（device_type）→ 权重20
3. 设备名称（device_name）→ 权重1
4. 规格型号（spec_model）→ 权重5
5. 关键参数（key_params）→ 权重15

#### 3. 权重分配规则

设备录入阶段使用简单直接的权重分配：
1. 检查是否来自device_type字段 → 20.0
2. 检查是否来自key_params → 15.0
3. 检查是否来自brand字段 → 10.0
4. 检查是否来自spec_model字段 → 5.0
5. 其他特征（device_name等） → 1.0

**不使用的判断逻辑**（仅在匹配阶段使用）：
- ❌ 设备类型关键词判断
- ❌ 品牌关键词判断
- ❌ 型号模式匹配
- ❌ 重要参数判断
